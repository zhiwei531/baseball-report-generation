# Asset Provenance

This document maps the final `vicon_2026_julian_coach 4` report schema to the builder that creates each asset family.

For the exact image-by-image checklist in final report order, see `docs/FINAL_REPORT_IMAGE_CHECKLIST.md`.

## Current Entry

The supported report-generation entry is:

```bash
python scripts/report_cli.py --config configs/default_report_pipeline.json
```

It reads shared defaults from `configs/default_report_pipeline.json`. For a new player, copy that config, update the path and identity fields, then run:

```bash
python scripts/report_cli.py \
  --config configs/<player_slug>_report_pipeline.json
```

Individual builder commands below document provenance and are useful for debugging, but the config-driven wrapper is the production path.

## Scope

Implemented here:

- Batting metrics CSV builders.
- Batting 3D reconstruction GIF/PNG builders.
- Batting 2D overlay annotation builder.
- Batting frontend illustration annotation builder.
- Batting kinetic-chain and analyst chart builders.
- Final batting HTML schema polish pass.

Built separately by the pitching pipeline:

- Pitching report sections and `pitch_assets/*`.
- Pitching 2D overlay images, pitching metric illustrations, pitching kinetic-chain charts, and pitching Vicon event GIFs.

Pitching assets are prepared outside the report CLI and referenced by `pitch_report` in the report config.

## Final HTML Builder

Production wrapper:

```bash
python scripts/report_cli.py --config configs/default_report_pipeline.json
```

Lower-level builders called by the wrapper:

- `scripts/build_julian_coach_metrics_section.py`
- `scripts/apply_batting_coach_values.py`

`build_julian_coach_metrics_section.py` creates the standalone final-schema HTML and imports the optional pitching HTML. `apply_batting_coach_values.py` refreshes final-schema batting values, peer comparisons, and researcher chart assets.

Production command:

```bash

```

Equivalent lower-level command:

```bash

  --input-dir ../vicon_2026 \
```

## Batting Body Metrics XLSX

The production wrapper runs the XLSX stage unless `--skip-xlsx` is passed. Lower-level workbook builder:

```bash
METRICS_PATH=reports/vicon_2026_julian_coach/batting_dashboard_metrics.csv \
OUT_DIR=outputs/batting_metrics_excel \
SAMPLE_NAME=julian \
node scripts/build_batting_metrics_xlsx.mjs
```

Production wrapper:

```bash
python scripts/report_cli.py --config configs/default_report_pipeline.json
```

Output:

```text
outputs/batting_metrics_excel/<source_c3d_basename>_batting_report_metrics.xlsx
outputs/batting_metrics_excel/<source_c3d_basename>_batting_报告指标.png
outputs/batting_metrics_excel/<source_c3d_basename>_batting_事件定位.png
outputs/batting_metrics_excel/<source_c3d_basename>_batting_说明.png
```

Workbook sheets:

- `报告指标`: frontend metric, backend metric key, value, unit, Vicon source, explanation.
- `事件定位`: Ready/Contact frames, swing segment, 3D reconstruction window.
- `说明`: C3D source, assumptions, frame rate, limitations, and formula notes.

Runtime note: this script uses `@oai/artifact-tool`, available in the Codex/OpenAI document runtime. Outside that runtime, replace this script with a local XLSX writer or pass `--skip-xlsx` to the staged pipeline.

The alignment script uses MediaPipe Tasks Heavy 33-point landmarks and imports the C3D parser/event logic from `build_vicon_2026_metrics.py`. For the validated Julian trial, C3D event `bat_speed_peak` maps Vicon frame `854` to reviewed video frame `184`; encoded playback FPS is `29.48022763100522`, capture FPS is `240`, and the slow factor is `8.141049757281554`.

## Pitching Assets: Separate Pipeline

The final schema contains pitching sections and references assets under:

```text
pitch_assets/
  vicon_reconstruction_events/
  vicon_2d_geometry_annotations/
  frontend_metric_illustrations_pitch/
  frontend_metric_illustrations_pitch_event/
  kinetic_chain/
  analyst_charts/
  lineart_actions/
```

This repository generates those assets through the independent pitching command:

```bash
python scripts/pitching/build_pitch_template_metrics_report.py \
  --manifest configs/pitching/manifest.json \
  --template-dir reports/pitching_template \
  --out-dir reports/pitching
```

The batting staged pipeline accepts the built pitching report through the config field `pitch_report`:

```bash
python scripts/report_cli.py --config configs/<player_slug>_report_pipeline.json
```

The builder behavior is:

1. Read pitching sections from `path/to/pitching/index.html`.
2. Copy `path/to/pitching/assets/` into the batting report output as `pitch_assets/`.
3. Rewrite pitching section asset paths from `assets/...` to `pitch_assets/...`.
4. Embed the pitching sections into the final combined schema.

The pitching build should guarantee that `index.html` and sibling `assets/` contain the directories listed above. Keep `--out-dir` separate from the batting `report_dir`; the batting builder copies pitching assets into its own `pitch_assets/` folder during integration.
